eu.hydrologis.jgrass.models.openmimodel

Identifier:
eu.hydrologis.jgrass.models.openmimodel

Since:
[Enter the first release in which this extension point appears.]

Description:
This extension point creates an openmi based model that is compatible with the JGrass console engine and Jgrass modeling language for chaining environmental models. For informations have a look at: www.jgrass.org www.openmi.org

Configuration Markup:

<!ELEMENT extension (model*)+>

<!ATTLIST extension

point CDATA #REQUIRED

id    CDATA #IMPLIED

name  CDATA #IMPLIED

>


<!ELEMENT model (EMPTY?)>

<!ATTLIST model

id          CDATA #REQUIRED

name        CDATA #REQUIRED

icon        CDATA #IMPLIED

class       CDATA "enter new class here"

inputitems  CDATA "null"

outputitems CDATA "null"

guixml      CDATA #IMPLIED

>

  • id -
  • name -
  • icon -
  • class -

    The class name for the new model to be created.

    Note that in order to follow JGrass's guidelines the class name and package should be chosen following the schema: to create a r.something command create a package eu.hydrologis.jgrass.models.r.something and put inside there a class named r_something.java

    Note that the class will need to have the following constructors defined, to be functional:

        public h_something() {
            super();
            err = FluidUtils.newPrintStream(null, System.err);
            out = FluidUtils.newPrintStream(null, System.out);
        }

        public h_something( PrintStream output, PrintStream error ) {
            super();
            err = FluidUtils.newPrintStream(error, System.err);
            out = FluidUtils.newPrintStream(output, System.out);
        }
  • inputitems -

    A comma separated list of input exchange items that the model will accept.

    For example if it has two input exchange items that may be linked to the model, the entry could be kind of:

    map=0,rain=1
  • outputitems -

    A comma separated list of output exchange items that the model will accept.

    For example if it has one output exchange items that may be linked to the model, the entry could be kind of:

    discharge=0
  • guixml -

    A correct xml file content that can represent the command as off the uibuilder guidelines. This string will be correctly placed in the system as a file and taken as gui definition for the command.

    For example for h.pitfiller the gui definition is:

    <command descr="h.pitfiller" name="h.pitfiller">
        <field desc="in" name="input" order="0" repr="--igrass-elevation #" required="true" type="rastermap"/>
        <field desc="out" name="output" order="1" repr="--ograss-pit #" required="true" type="string"/>
        <i18n key="in" en="input elevation map" it="mappa di elevazione in ingresso"/>
        <i18n key="out" en="depitted output map" it="mappa depittata in uscita"/>
    </command>
